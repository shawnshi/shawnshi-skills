# 创意与需求收敛

## 使用边界

只在用户明确要求脑暴、设计或评估想法时使用，不把所有编码任务自动改造成长时间的前置讨论。默认只分析和形成设计，不编写生产代码；用户随后要求实现时再进入实现流程。

## 收敛问题

1. 用一句话重述用户、痛点和预期结果。
2. 区分已知事实、假设、偏好和不可改变的约束。
3. 找出最可能改变架构或范围的未知项。
4. 只有未知项会阻塞方案选择时才提问；每轮优先问一个关键问题，已有上下文足够时直接继续。
5. 把过大的需求切成可独立验证的核心闭环和后续扩展。

不要自动读取历史设计、私人记忆或外部知识库。只有用户明确要求复用既有决策并授权相应数据源时才检索。

## 生成并压力测试方案

1. 提出 2–3 个有实质差异的方案，避免只改变命名。
2. 对每个方案说明：
   - 核心机制与边界；
   - 依赖和不可逆决策；
   - 成本、复杂度和失败模式；
   - 最小验证实验；
   - 适用和不适用条件。
3. 从用户目标出发给出一个推荐，并说明什么新证据会改变推荐。
4. 检查单点故障、规模增长、数据契约、权限、安全、回滚和运维负担；只展开与当前问题相关的项目。

复杂任务可以在行业研究、技术可行性和风险审查彼此独立且当前环境支持时选用子代理。需要联网获取当前市场或技术信息时先取得用户授权。

## 交付

默认输出：

1. 问题定义；
2. 范围内与范围外；
3. 关键假设；
4. 方案对比；
5. 推荐方案；
6. 最小验证步骤；
7. 开放问题和停止条件。

只有依赖关系复杂时才加入 Mermaid。保存为设计文档、写入 plans 目录或更新长期记忆前，必须取得明确授权并确认路径。不要自动清除用户保留的 TODO 或 TBD；把未决项作为决策债务列出。
