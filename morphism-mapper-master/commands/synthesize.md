# /morphism-synthesize 命令

## 用途
执行拉回操作（Pull-back），将 Domain B 的定理映射回 Domain A，生成可执行的提案。

## 用法
```
/morphism-synthesize
```

**注意**: 此命令依赖于当前会话中已完成的映射。如未进行映射，会先提示选择 Domain B。

## 输出格式
```markdown
### 【推演提案】

#### 方案一：承载力管理策略
**来源定理**: 生态学 - 承载力与种群调节
> 种群增长受环境承载力限制。当种群接近承载力时，密度制约因素会抑制增长。

**映射逻辑**:
- 用户流失不是因为产品不好，而是因为市场承载力达到上限
- 继续追求增长会导致"过度放牧"，损害现有用户体验
- 应将策略从"获取新用户"转向"优化现有用户生态"

**可执行方案**:
1. 停止大规模获客投放，将预算转向现有用户服务
2. 识别并保护"关键种"用户（高影响力、高活跃度的核心用户）
3. 设计用户分层体系，为不同生态位提供差异化价值

**预期效果**: 用户留存率提升 15-20%，用户生命周期价值提升 30%

**验证方式**:
- 监测留存率变化
- 追踪 NPS 评分
- 观察核心用户活跃度

---

#### 方案二：边缘效应利用
**来源定理**: 生态学 - 边缘效应与生态交错带
> 两种生态系统的交界处往往具有最高的生物多样性和生产力。

**映射逻辑**:
- 纯工具型产品或纯社交型产品都有明确边界
- 处于工具+社交交界地带的产品具有独特优势
- 用户在产品中的"社交边缘行为"被忽视了

**可执行方案**:
1. 识别用户在使用产品中自然产生的协作行为
2. 将这些边缘行为产品化，创建轻量级社交功能
3. 不追求成为社交平台，而是成为"有社交属性的工具"

**预期效果**: 打开新的增长空间，差异化竞争优势

**验证方式**:
- 边缘功能的使用率
- 用户自发分享行为
- 竞品模仿难度
```

## 执行步骤

1. 检查当前会话的映射状态
2. 从 Domain B 的 Theorems 中选择最相关的 3 个
3. 对每个定理执行 Pull-back 映射
4. 生成具体的可执行方案
5. 为每个方案提供验证方式

## 注意事项
- 提案必须具体可执行，不能停留在概念层面
- 每个方案都需要说明来源定理和映射逻辑
- 必须包含"不可直视"的洞察——正常思维想不到的角度
