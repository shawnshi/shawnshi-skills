# Few-Shot 示例

本文件包含 Morphism Mapper 的典型使用示例，用于指导 Agent 的学习。

---

## 示例 1：ETF 用户留存问题

**用户输入**: "我想设计 ETF 产品的用户留存体系，目前用户流失严重。"

### Phase 1: Category Extraction

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 用户 | 基金投资者，具有情绪波动 |
| Object | 资金 | 投资本金，追求增值 |
| Object | 净值曲线 | 产品表现的可视化 |
| Morphism | 交易 | 买入/卖出行为 |
| Morphism | 持有 | 长期投资状态 |
| Morphism | 查看 | 观察净值变化 |
| Constraint | 波动 | 市场不确定性 |
| Constraint | 损失厌恶 | 心理学偏见 |

### Phase 2: Domain Selection

**选择**: zhuangzi（庄子哲学）

**理由**: 问题的核心在于用户对短期波动的过度反应（损失厌恶），需要改变观察尺度和认知框架。

### Phase 3: Functorial Mapping

| Domain A | 映射关系 | Domain B |
|----------|----------|----------|
| 用户 | ≅ | 观察者 |
| 净值曲线 | ≅ | 所见之物 |
| 短期波动 | ≅ | 朝菌/蟪蛄之见 |
| 长期趋势 | ≅ | 冥灵/大椿之见 |
| 查看行为 | → | 观 |
| 情绪反应 | → | 惑 |

### Phase 4: Pull-back & Synthesis

**来源定理**: 尺度相对论 - "小知不及大知，小年不及大年"

**提案**:

开发"冥灵模式"UI：
- 将净值曲线的最小观测单位设为"季度"而非"日"
- 默认展示 3 年/5 年视角，而非 1 个月
- 引入"时间胶囊"功能：投资后锁定查看权限，到期才能打开
- 消除"蟪蛄之智"引发的短期恐慌

## 示例 2：产品功能冗余

**用户输入**: "我们的产品功能越来越多，用户反而更困惑了，怎么办？"

### Phase 1: Category Extraction

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 功能 | 产品的具体能力 |
| Object | 用户 | 功能的使用者 |
| Object | 界面 | 功能的呈现方式 |
| Morphism | 添加 | 持续增加新功能 |
| Morphism | 使用 | 用户与功能的交互 |
| Morphism | 学习 | 用户理解功能的过程 |
| Constraint | 认知负荷 | 人类工作记忆有限 |
| Constraint | 选择悖论 | 选项过多导致决策困难 |

### Phase 2: Domain Selection

**选择**: thermodynamics（热力学）

**理由**: 功能增加导致系统复杂度（熵）增加，需要理解熵增原理和耗散结构。

### Phase 3: Functorial Mapping

| Domain A | 映射关系 | Domain B |
|----------|----------|----------|
| 功能 | ≅ | 能量/物质 |
| 产品 | ≅ | 系统 |
| 用户认知 | ≅ | 有序度 |
| 功能添加 | → | 能量输入 |
| 用户困惑 | → | 熵增/无序 |
| 功能整合 | → | 做功/有序化 |

### Phase 4: Pull-back & Synthesis

**来源定理**: 热力学第二定律 - 孤立系统的熵永不减少

**提案**:

1. **功能熵减策略**
   - 不是停止添加功能，而是为每个新功能移除一个旧功能
   - 保持"功能熵"恒定或降低

2. **开放系统设计**
   - 将产品从封闭系统转变为开放系统
   - 允许用户自定义功能组合（插件化）
   - 让用户自己决定什么是"有用"的

3. **相变临界点**
   - 识别功能数量的"相变点"
   - 在临界点进行功能重组，而非渐进添加

## 示例 3：团队协作低效

**用户输入**: "我们团队人不少，但协作效率很低，信息传递总是出问题。"

### Phase 1: Category Extraction

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 团队成员 | 信息的生产者和消费者 |
| Object | 信息 | 需要传递的内容 |
| Object | 任务 | 协作的目标 |
| Morphism | 沟通 | 信息在成员间流动 |
| Morphism | 决策 | 基于信息的判断 |
| Morphism | 执行 | 将决策转化为行动 |
| Constraint | 信息损耗 | 传递过程中的失真 |
| Constraint | 延迟 | 信息传递的时间成本 |

### Phase 2: Domain Selection

**选择**: network_theory（网络理论）

**理由**: 团队本质上是一个信息网络，协作效率取决于网络结构。

### Phase 3: Functorial Mapping

| Domain A | 映射关系 | Domain B |
|----------|----------|----------|
| 团队成员 | ≅ | 节点 |
| 沟通渠道 | ≅ | 边/连接 |
| 信息 | ≅ | 信号/数据包 |
| 协作效率 | ≅ | 网络传输效率 |
| 信息损耗 | → | 噪声/丢包 |
| 关键成员 | → | 枢纽节点 |

### Phase 4: Pull-back & Synthesis

**来源定理**: 弱连接的力量 - 弱连接在信息传播中比强连接更重要

**提案**:

1. **弱连接强化计划**
   - 团队内部的强连接（日常协作）已经饱和
   - 问题可能在于缺乏跨团队、跨部门的弱连接
   - 建立"随机咖啡"机制：每周随机匹配不同成员一对一交流

2. **枢纽节点保护**
   - 识别团队中的"信息枢纽"（通常是资深成员或跨职能角色）
   - 减轻他们的执行负担，保护其信息整合功能
   - 为他们提供信息处理工具，而非更多执行任务

3. **小世界网络设计**
   - 当前可能是高度 clustered 的网络（小团体）
   - 引入少数"桥接者"角色，连接不同子团队
   - 不需要每个人都与所有人连接，保持"六度分隔"即可

## 示例 4：品牌战略迷茫

**用户输入**: "我们的品牌定位不清晰，用户记不住我们，营销效果很差。"

### Phase 1: Category Extraction

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 品牌 | 企业的符号化表达 |
| Object | 用户 | 品牌的感知者 |
| Object | 竞品 | 市场上的其他品牌 |
| Morphism | 传播 | 品牌信息的扩散 |
| Morphism | 认知 | 用户对品牌的理解 |
| Morphism | 记忆 | 品牌印象的留存 |
| Constraint | 注意力稀缺 | 用户心智资源有限 |
| Constraint | 同质化 | 竞品信息噪音 |

### Phase 2: Domain Selection

**选择**: mythology（神话学/原型心理学）

**理由**: 品牌的本质是叙事，最强大的品牌都激活了集体无意识中的原型。

### Phase 3: Functorial Mapping

| Domain A | 映射关系 | Domain B |
|----------|----------|----------|
| 品牌 | ≅ | 英雄/导师/其他原型 |
| 用户 | ≅ | 旅程中的英雄 |
| 产品 | ≅ | 魔法道具/武器 |
| 品牌传播 | → | 召唤/启程 |
| 用户购买 | → | 接受挑战/获得道具 |
| 品牌忠诚 | → | 归来/传承 |

### Phase 4: Pull-back & Synthesis

**来源定理**: 英雄之旅的单一神话 - 所有文化的英雄故事都遵循同一基本结构

**提案**:

1. **原型定位**
   - 不是定义"我们是什么公司"，而是"我们激活什么原型"
   - 选择：照顾者（Caregiver）、创造者（Creator）、探索者（Explorer）、英雄（Hero）等
   - 所有品牌决策都围绕原型一致性

2. **用户英雄化**
   - 品牌不是英雄，用户才是英雄
   - 品牌是导师（Yoda）或魔法道具（ lightsaber）
   - 营销内容展示"用户在使用产品后的转变"，而非产品功能

3. **阈限体验设计**
   - 购买/注册不是终点，而是"阈限"——从旧身份到新身份的过渡
   - 设计 onboarding 为新英雄的"试炼"
   - 庆祝用户的"归来"（成就系统、用户故事分享）

## 示例 5：定价策略困境

**用户输入**: "我们的定价总是被竞品牵着鼻子走，陷入价格战，利润越来越薄。"

### Phase 1: Category Extraction

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 我方 | 定价决策者 |
| Object | 竞品 | 价格跟随者/领导者 |
| Object | 用户 | 价格敏感/不敏感的购买者 |
| Morphism | 定价 | 价格决策行为 |
| Morphism | 跟随 | 对竞品价格的反应 |
| Morphism | 购买 | 用户的选择行为 |
| Constraint | 价格透明 | 信息对称 |
| Constraint | 利润压力 | 成本结构限制 |

### Phase 2: Domain Selection

**选择**: game_theory（博弈论）

**理由**: 定价是典型的多人策略互动，可以用博弈论分析均衡和最优策略。

### Phase 3: Functorial Mapping

| Domain A | 映射关系 | Domain B |
|----------|----------|----------|
| 我方 | ≅ | 玩家1 |
| 竞品 | ≅ | 玩家2...n |
| 定价策略 | ≅ | 策略空间 |
| 利润 | ≅ | 收益函数 |
| 价格战 | → | 囚徒困境 |
| 跟随定价 | → | 纳什均衡 |

### Phase 4: Pull-back & Synthesis

**来源定理**: 承诺的价值 - 通过限制自己的未来选择，可以改变他人的预期

**提案**:

1. **价格承诺策略**
   - 公开承诺"永不降价"或"最低价保证"
   - 这不是限制自己，而是限制对手的选择空间
   - 让对手预期到降价不会赢得市场份额，从而放弃价格战

2. **混合策略定价**
   - 可预测的价格会被对手针对性应对
   - 引入随机化：定期推出限时优惠，但时间和幅度不可预测
   - 让对手无法确定你的下一步

3. **改变博弈结构**
   - 当前是价格博弈，转为价值博弈
   - 引入新的维度（服务、体验、社区），使价格不再是唯一比较维度
   - 将零和博弈转变为正和博弈
