# 颜色修复规则（深色底 ↔ 浅色底转换）

## ⚠️ 修改守则（必读）

> **不要删改已验证的规则，只能在后面叠加。**
>
> 每次遇到新问题时的流程：
> 1. **先诊断**：写诊断脚本，搞清楚是哪类形状、填充是什么、为什么现在的规则不对
> 2. **记录根因**：在下方"踩坑经验"里写明这个场景的特征
> 3. **最小改动**：只修改最精准的那一个判断，不重写整个函数
> 4. **验证不回归**：改完后跑之前已知的好案例，确认没有破坏
>
> 当前核心逻辑（`_fix_shape_text_colors_smart`，已稳定，**不要轻易改动**）：
> - 形状有显式深色填充 → 保留白字（只修渐变）
> - 透明形状 + 与**同 GROUP 内兄弟**深色形状重叠 ≥ 65% → 保留白字（叠在深色面板上）
> - 其他所有情况 → 改为深色
>
> 这个逻辑经历了 5 次迭代才稳定，**核心是"兄弟深色检测"而不是"全局坐标检测"**。
> 全局坐标检测在复杂页面（如含 43 个深色格子的架构图）会大量误判。

| 规则 | 覆盖场景 |
|------|---------|
| 透明/无填充形状中，非近白 `prstClr` → `#FFFFFF` | 黑色、蓝色等预设文字色 |

## 已实现规则（深色底 → 浅色底）

| 规则 | 覆盖场景 |
|------|---------|
| `prstClr white` 等预设浅色 → `#262626` | 明确的白色预设 |
| `srgbClr` 三通道≥200 → `#262626` | 直接指定近白色 |
| `srgbClr + lumMod/lumOff` 计算实际亮度 | `5B9BD5+lumOff=95000`→近白 |
| `schemeClr bg1/lt1/bg2/lt2` → `#262626` | 主题背景/亮色槽位（含带修饰符版本） |
| `schemeClr 任意 + lumOff≥40000` → `#262626` | accent5 等+高 lumOff→近白 |
| 文字 `gradFill` 全浅色色标 → 蓝青渐变 | 深色底的渐变效果字 |
| 深色背景形状：保留实色白字，仍修渐变文字 | badge/按钮里的渐变近白字 |
| 渐变 alpha 平均 < 50% → 视为透明，修文字 | 半透明蓝色渐变背景（P50类型） |
| GROUP 内深色子形状叠加检测 → 白字保留 | 透明文字框叠在 GROUP 内深色面板上（P49类型） |
| 大面积/中小面积深色实色形状 → 加透明度 | 深色底座/圆盘在浅色底过于突兀（P6类型） |
| 显式浅色实心填充形状 → 透明 | 深色底的色块背景 |

## 关键可配置常量（engine/assemble_template.py 顶部）

```python
_DARK_HEX = '262626'                              # 深灰，替换浅色实色文字
_GRADIENT_FALLBACK_COLORS = ['0165FF', '00B8F3']  # 渐变文字替换色（蓝→青蓝）
_LARGE_DARK_AREA_THRESHOLD = 0.008                # 深色形状透明化：面积 ≥ 0.8% 触发
_LARGE_DARK_ALPHA = 15000                         # 透明化后的不透明度（15%，调大=更深）
```

## 踩坑经验

**坑1：GROUP 子形状不在顶层 `slide.shapes` 里**
凡是需要扫描 slide 全部形状的逻辑（建深色区域表、透明化大色块），都必须递归进 GROUP，否则 GROUP 内的形状被跳过。

**坑2：渐变 alpha 透明度被忽略**
`_shape_has_dark_fill` 原先只看颜色亮度，alpha=30% 的蓝色会被误判为深色，导致白字保留在近透明背景上不可见。

**坑3：GROUP 子形状坐标是 slide 绝对坐标**
python-pptx 里 GROUP 子形状的 `.left/.top` 直接是幻灯片绝对坐标，不需要加 GROUP 偏移量换算。

**坑4：_shape_on_dark_region 位置检测导致规则冲突（已废弃）**
曾为处理"透明文字框叠在深色面板上（P49类型）"加入了 `_shape_on_dark_region`，用 30% 坐标重叠判断形状是否在深色背景上。
但在含大量深色小格子的复杂架构图页面（如信创架构图，43 个深色区域），几乎所有透明标题文字框都误判为"叠在深色区域上"，导致白字全部被错误保留。
**结论：规则越加越复杂会互相打架。已回归最简逻辑：`_fix_shape_text_colors_smart` 只看 `_shape_has_dark_fill`——有深色填充保白，其余全改黑。若 P49 类型将来再遇到，需用更精准的方式（如 z-order 判断）而不是坐标重叠 heuristic。**

## 待解决缺口

- 黄色文字 `FFFF00`：在白底不可见，但不满足"近白"判断，需要低对比度规则
- 大面积渐变填充深色形状（如 `sysClr=window → srgbClr=00175A`）：暂未透明化
