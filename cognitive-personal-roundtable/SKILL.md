# 认知圆桌

## 设计圆桌

1. 明确议题、决策目标、事实基线和希望挑战的假设。
2. 选择 3–5 个能引入不同变量的分析视角，例如实践、学术、财务、伦理、历史或反方。
3. 角色是分析镜头，不是权威替身。使用真实人物时，不编造其原话、立场或私人动机；必要时改用“某思想传统的视角”。
4. 至少设置一个有根据的反方或意外视角，但不要为了冲突故意极端化。
5. 需要人物选择参考时读取 [references/personas.md](references/personas.md)。

## 进行交锋

按三轮组织：

1. 立场：每个视角给出主张、证据和关键变量。
2. 质询：明确指出其他视角忽略的前提、代价或反例。
3. 修正：说明哪些立场经质询后改变，哪些分歧仍无法消除。

每段发言都应增加信息，不重复同一结论。不能找到证据时标记为假设。

常规圆桌由主代理直接完成。只有视角能够独立研究、材料量大且当前环境支持时，才可用子代理；只要求其返回立场、证据、反例和不确定性，不要求隐藏推理。主代理负责事实核对和最终综合。

## 综合

建立张力图：

- 已确认的共同事实；
- 直接冲突的判断；
- 导致分歧的底层假设；
- 所有视角都遗漏的变量；
- 哪项新证据能解决分歧。

不要强行制造共识，也不要把争论强度当作结论质量。

## 交付

默认输出：

1. 议题和事实基线；
2. 各视角的核心立场；
3. 关键交锋；
4. 共识、分歧和遗漏变量；
5. 面向用户的决策选项与验证问题。

只有实际生成了多个分片文件时，才考虑使用 [scripts/merger.py](scripts/merger.py) 合并。默认不生成分片、不写入知识库。保存归档、更新记忆或其他持久化操作必须由用户明确授权。
