## 一、信创政策背景

### 政策驱动力
- **2022-2027 信创三期规划**：8 大重点行业（含医疗卫生）分阶段推进国产化替代。
- **等保 2.0**：三级以上系统必须满足网络安全等级保护标准。
- **数据安全法 & PIPL**：医疗数据属于重要数据，跨境传输受严格限制。
- **医疗行业优先级**：三甲医院为第一批信创替代目标，10+ 个省份已发文推动。

### 医院信创替代范围

| 替代层级 | 核心替代对象 | 替代紧迫度 |
|:---|:---|:---:|
| 芯片 | x86 → 国产 CPU | ⭐⭐⭐ |
| 操作系统 | Windows Server / CentOS → 国产 Linux | ⭐⭐⭐⭐ |
| 数据库 | Oracle / SQL Server → 国产数据库 | ⭐⭐⭐⭐⭐ |
| 中间件 | WebLogic / Tomcat → 国产中间件 | ⭐⭐⭐ |
| 办公应用 | MS Office → WPS / 永中 | ⭐⭐ |

## 四、数据库适配矩阵（重点）

> 数据库是信创替代的 **最高风险环节**，也是 HIS/EMR 迁移的核心瓶颈。

| 数据库 | 兼容性 | 性能定位 | 医疗典型案例 | 迁移难度 |
|:---|:---|:---|:---|:---:|
| 达梦 (DM8) | Oracle 高度兼容 | OLTP + OLAP | 多家三甲医院 HIS 替换 | ⭐⭐⭐ |
| 人大金仓 (KingbaseES) | PostgreSQL 兼容 | OLTP | 医院信息集成平台 | ⭐⭐ |
| 南大通用 (GBase) | 多模型 | OLAP / 分析型 | 数据中台/BI 分析 | ⭐⭐ |
| 瀚高 (HighGo) | PostgreSQL 兼容 | OLTP | 中小型医院 | ⭐⭐ |
| OceanBase | MySQL 兼容 | 分布式 OLTP | 高并发互联网医院 | ⭐⭐⭐⭐ |
| TiDB | MySQL 兼容 | 分布式 HTAP | 大规模数据分析 | ⭐⭐⭐ |

### 数据库迁移核心风险

| 风险 | 说明 | 减缓策略 |
|:---|:---|:---|
| 存储过程不兼容 | Oracle PL/SQL → DM/PG 语法差异 | 分阶段迁移 + 自动化转换工具 |
| 性能回退 | 复杂查询优化器差异 | 压力测试 + SQL 调优 + 索引重建 |
| 字符集/排序差异 | 中文排序、时区处理不一致 | 统一字符集标准、回归测试覆盖 |
| 第三方驱动不兼容 | HIS 中间件/报表工具驱动 | 确认厂商适配清单、备选方案 |

## 六、信创部署架构建议

### 分阶段替代策略（推荐）

```
阶段一 (0-6月)：外围系统信创化
├── OA/邮件/文档系统 → UOS + WPS
├── 数据报表/BI → 国产数据库 + 帆软
└── 新建子系统直接信创部署

阶段二 (6-18月)：核心数据层信创化
├── 数据库迁移 Oracle → 达梦/金仓
├── 集成平台中间件替换
└── 双轨并行运行（新旧数据库同步）

阶段三 (18-36月)：全栈信创化
├── HIS/EMR 核心系统迁移至信创环境
├── 芯片层逐步替换（服务器换代时）
└── 旧系统完全下线
```

### 关键成功要素

1. **双轨并行**：信创环境与原有环境并行运行，确保业务连续性
2. **压力测试**：核心业务系统在信创环境下必须完成全量压力测试
3. **应急回退**：制定完整的回退预案，确保信创迁移失败时可快速恢复
4. **厂商协同**：HIS 厂商 + 信创厂商联合适配测试
5. **人员培训**：IT 运维团队需提前完成信创环境运维培训
