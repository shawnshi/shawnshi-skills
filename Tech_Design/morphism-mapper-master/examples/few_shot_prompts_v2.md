## 示例 1：ETF 用户留存问题 (基础四阶段 + Context Anchoring + Commutativity Check)

**用户输入**: "我想设计 ETF 产品的用户留存体系，目前用户流失严重。"

### Phase 0: Context Anchoring ⚓

- **Identity**: 基金公司产品总监（推断：关注 KPI、合规、规模、股东回报）
- **Resources**: 拥有交易数据、App 运营位、客服团队（推断：技术资源充足，但受监管约束）
- **Constraints**: 
  - 🔴 监管红线：不能承诺收益、不能诱导频繁交易
  - 🟡 市场约束：无法控制 ETF 净值波动
  - 🟢 时间窗口：季度考核压力

*Context Injection*: 所有方案必须标注合规风险等级，避免"保证收益"等违规表述。

### 【范畴骨架】- Domain A

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 用户 | 基金投资者，具有情绪波动 |
| Object | 资金 | 投资本金，追求增值 |
| Object | 净值曲线 | 产品表现的可视化 |
| Morphism | 交易 | 买入/卖出行为 |
| Morphism | 持有 | 长期投资状态 |
| Morphism | 查看 | 观察净值变化 |
| Constraint | 波动 | 市场不确定性 |
| Constraint | 损失厌恶 | 心理学偏见 |

### 【异构域】- Domain B: zhuangzi (庄子哲学)

**选择理由**: 问题的核心在于用户对短期波动的过度反应（损失厌恶），需要改变观察尺度和认知框架。庄子哲学的"尺度相对论"完美匹配此问题。

**Domain B 核心结构** (来自 zhuangzi_v2.md):

**Core Objects**:
- 观察者 (Observer): 认知主体，决定所见
- 所知之物 (Known): 被观察的现象
- 时间尺度 (Timescale): 冥灵/大椿/朝菌的认知差异
- 相对性 (Relativity): 大小、长短、是非的相对

**Core Morphisms**:
- 观 (Observing): 观察行为本身决定所见
- 化 (Transforming): 物化与自我转化
- 忘 (Forgetting): 剥离执念，回归本真

**关键定理**:
- **定理 1: 尺度相对论** - "小知不及大知，小年不及大年"
  - Applicable_Structure: 存在时间尺度差异的认知系统
  - Mapping_Hint: 当 Domain A 面临短期波动引发的焦虑时，识别时间尺度作为核心变量，通过切换观察尺度实现认知重构
  - Case_Study: 朝菌不知晦朔，蟪蛄不知春秋 vs 冥灵五百岁为春

### 【映射矩阵】

| Domain A | 映射关系 | Domain B | 同构性验证 |
|----------|----------|----------|------------|
| 用户 | ≅ | 观察者 | 都是认知主体，决定如何解读现象 |
| 净值曲线 | ≅ | 所见之物 | 都是被观察的现象 |
| 短期波动 | ≅ | 朝菌/蟪蛄之见 | 短时间尺度的焦虑认知 |
| 长期趋势 | ≅ | 冥灵/大椿之见 | 长时间尺度的宏观认知 |
| 查看行为 | → | 观 | 观察行为塑造所见 |
| 情绪反应 | → | 惑 | 执念引发的痛苦 |

### 【推演提案】

**方案标题**: "冥灵模式"用户留存体系

**来源定理**: 庄子 - 尺度相对论

**映射逻辑**: 用户的流失源于"朝菌之见"——以日为单位观察净值，被短期波动折磨。通过改变时间尺度，将用户的"观"从短期切换到长期，将"惑"转化为"悟"。

**可执行方案**:

1. **时间尺度重构 UI**
   - 将净值曲线的最小观测单位设为"季度"而非"日"
   - 默认展示 3 年/5 年视角，而非 1 个月
   - 隐藏日净值波动，突出年化收益率

2. **"时间胶囊"功能**
   - 投资后可选择锁定查看权限
   - 设置解锁日期（如1年后、3年后）
   - 消除"蟪蛄之智"引发的短期恐慌

3. **认知框架教育**
   - 用户引导：对比"朝菌"vs"大椿"的投资故事
   - 案例展示：长期持有 vs 频繁交易的收益对比
   - 社区建设："冥灵圈"——长期投资者的交流平台

**预期效果**:
- 短期查看行为减少 60%
- 用户持仓周期延长 2-3 倍
- 流失率降低 40%

**验证方式**:
- A/B 测试：对照组 vs "冥灵模式"组
- 追踪指标：平均持仓天数、查看频率、赎回率

### Phase 4.1: Commutativity Check ⚠️

在生成最终提案前，验证映射的逆运算合理性：

**正向路径验证**：
- ETF 用户流失问题 → 庄子"尺度相对论" → "冥灵模式"UI方案
- 映射逻辑：短期波动焦虑 → 改变观察尺度 → 长期视角缓解焦虑 ✓

**逆向路径验证**（关键）：

| 公理检查 | 验证内容 | 结果 |
|----------|----------|------|
| **法律公理** | 隐藏日涨跌幅是否违规？是否构成"诱导长期持有"？ | ⚠️ 需法务确认：仅调整默认展示，不禁止查看，风险可控 |
| **伦理公理** | 是否侵犯用户知情权？ | ✅ 用户可主动选择查看日净值，不构成欺骗 |
| **经济公理** | 长期持有是否一定对投资者有利？ | ⚠️ 需加注："长期持有"不承诺收益，仅作为行为干预手段 |

**结构守恒检查**：
- ✅ 映射未篡改：B中的"尺度相对论"映射回A仍是"改变观察方式"，结构一致
- ✅ 无范畴错误：未将"认知干预"误映射为"收益保证"

**结论**：✅ **逻辑闭合，但需在落地时加注风险提示**

**合规适配建议**：
- 在"时间胶囊"功能中加入明确提示："锁定查看不代表锁定期内一定盈利/亏损"
- 在App显著位置保留"日净值"查看入口（只是不默认展示）
- 所有宣传材料避免"长期投资一定能赚钱"等违规表述

## 示例 2：产品功能冗余 (含 Adjoint Balancer)

**用户输入**: "我们的产品功能越来越多，用户反而更困惑了，怎么办？"

### 【范畴骨架】- Domain A

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 功能 | 产品的具体能力 |
| Object | 用户 | 功能的使用者 |
| Object | 界面 | 功能的呈现方式 |
| Morphism | 添加 | 持续增加新功能 |
| Morphism | 使用 | 用户与功能的交互 |
| Morphism | 学习 | 用户理解功能的过程 |
| Constraint | 认知负荷 | 人类工作记忆有限 |
| Constraint | 选择悖论 | 选项过多导致决策困难 |

### 【异构域】- Domain B: thermodynamics (热力学)

**选择理由**: 功能增加导致系统复杂度（熵）增加，需要理解熵增原理和耗散结构。

**Domain B 核心结构** (来自 thermodynamics_v2.md):

**Core Objects**:
- 系统 (System): 被研究的对象边界
- 能量 (Energy): 做功的能力
- 熵 (Entropy): 系统的无序度

**Core Morphisms**:
- 能量输入 (Energy Input): 向系统注入能量
- 做功 (Work): 有序能量转化为有用功
- 热传递 (Heat Transfer): 能量以无序形式流动

**关键定理**:
- **定理 2: 热力学第二定律** - 孤立系统的熵永不减少
  - Applicable_Structure: 封闭系统，存在能量/信息输入
  - Mapping_Hint: 当 Domain A 面临复杂度增加导致的混乱时，识别系统边界是否封闭，通过开放系统或主动做功实现熵减
  - Case_Study: 生命体通过摄取负熵维持有序

### 【映射矩阵】

| Domain A | 映射关系 | Domain B | 同构性验证 |
|----------|----------|----------|------------|
| 功能 | ≅ | 能量/物质 | 都是系统的组成部分 |
| 产品 | ≅ | 系统 | 都是被研究的对象 |
| 用户认知 | ≅ | 有序度 | 认知越清晰 = 越有序 |
| 功能添加 | → | 能量输入 | 增加系统复杂度 |
| 用户困惑 | → | 熵增/无序 | 混乱度增加 |
| 功能整合 | → | 做功/有序化 | 消耗能量实现有序 |

### 【推演提案】

**方案标题**: 功能熵减与开放系统设计

**来源定理**: 热力学第二定律 + 耗散结构理论

**映射逻辑**: 产品是一个封闭系统，持续添加功能必然导致熵增（用户困惑）。解决方案：要么让系统开放（用户自定义），要么主动做功（功能整合）。

**可执行方案**:

1. **功能熵减策略**
   - 不是停止添加功能，而是为每个新功能移除一个旧功能
   - 保持"功能熵"恒定或降低
   - 每季度评估功能使用率，淘汰后 20%

2. **开放系统设计** (耗散结构)
   - 将产品从封闭系统转变为开放系统
   - 插件化架构：核心功能精简，扩展功能插件化
   - 让用户自己决定什么是"有用"的

3. **相变临界点管理**
   - 识别功能数量的"相变点"（如 7±2 个核心功能）
   - 在临界点进行功能重组，而非渐进添加
   - 分版本发布：精简版、专业版、企业版

**预期效果**:
- 新用户上手时间减少 50%
- 功能使用率提升 30%
- 用户满意度 NPS 提升 15 分

### 【伴随解】- Adjoint Balancer (可行性校验)

**成本-结构分析**:

| 方案 | 开发成本 | 用户学习成本 | 维护成本 | 可行性评分 |
|------|----------|--------------|----------|------------|
| 功能熵减 | 低 | 低 | 低 | ⭐⭐⭐⭐⭐ |
| 开放系统 | 高 | 中 | 高 | ⭐⭐⭐ |
| 相变管理 | 中 | 低 | 中 | ⭐⭐⭐⭐ |

**最优落地路径**:
1. **阶段 1** (立即执行): 功能熵减 - 零开发成本，立即见效
2. **阶段 2** (3个月后): 相变临界点管理 - 基于阶段1数据优化
3. **阶段 3** (6个月后): 评估开放系统 ROI，决定是否投入

**资源约束适配**:
- 小团队：只做功能熵减
- 中团队：功能熵减 + 相变管理
- 大团队：全方案实施

## 示例 3：SaaS 企业面临 AI 冲击 (含 Yoneda Probe)

**用户输入**: "我们是做传统 SaaS 的，ChatGPT 出来后客户流失严重，不知道怎么转型。"

### 【范畴骨架】- Domain A

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | SaaS 产品 | 传统软件服务 |
| Object | 客户 | 企业用户 |
| Object | AI 工具 | ChatGPT 等新技术 |
| Morphism | 订阅 | 客户付费使用 |
| Morphism | 替代 | AI 取代原有功能 |
| Morphism | 迁移 | 客户转向 AI |
| Constraint | 技术债务 | 传统架构难以改造 |
| Constraint | 收入下滑 | 客户流失导致 |

### 【Yoneda 拓扑画像】- 信息不透明时的补充分析

**触发条件**: 用户对 AI 冲击的具体机制描述模糊，需要反推核心对象定义。

**通过关系网反推**:
- SaaS 产品与 AI 工具的关系：功能重叠度？互补性？
- 客户迁移路径：完全替换 vs 混合使用？
- 技术债务约束：重构可能性 vs 重写成本？

**反推核心对象**:
- **核心价值** (Value Proposition): 客户真正购买的是什么？
  - 不是功能本身，而是**问题解决**
  - SaaS 解决的是"重复性工作自动化"
  - AI 解决的是"创造性工作辅助"

- **不可替代性** (Irreplaceability): 什么不会被 AI 取代？
  - 数据资产（客户历史数据）
  - 行业 know-how（垂直领域知识）
  - 工作流集成（企业已有流程）

### 【异构域】- Domain B: evolutionary_biology (进化生物学)

**选择理由**: SaaS 企业面临的环境剧变类似于物种面临的环境剧变，进化生物学的"适应辐射"和"关键创新"概念适用。

**Domain B 核心结构** (来自 evolutionary_biology_v2.md):

**Core Objects**:
- 物种 (Species): 生物分类单元
- 生态位 (Niche): 物种在生态系统中的角色
- 环境 (Environment): 选择压力来源

**Core Morphisms**:
- 选择 (Selection): 环境对性状的筛选
- 适应 (Adaptation): 物种对环境的调整
- 辐射 (Radiation): 快速分化占领新生态位

**关键定理**:
- **定理 5: 关键创新** - 某一性状的演化开启新的适应可能性
  - Applicable_Structure: 存在技术/能力突破导致新机会
  - Mapping_Hint: 当 Domain A 面临技术颠覆时，识别自身是否拥有可转化为关键创新的资产，通过重新定义价值主张实现生态位跳跃
  - Case_Study: 鸟类羽毛最初用于保温，后演化为飞行器官

### 【映射矩阵】

| Domain A | 映射关系 | Domain B | 同构性验证 |
|----------|----------|----------|------------|
| SaaS 产品 | ≅ | 物种 | 都是被选择的主体 |
| 市场环境 | ≅ | 生态环境 | 都施加选择压力 |
| 传统功能 | ≅ | 原有性状 | 都可能被淘汰 |
| AI 冲击 | → | 环境剧变 | 选择压力突变 |
| 客户流失 | → | 不适应者死亡 | 自然选择结果 |
| 转型 | → | 适应/辐射 | 生存策略 |

### 【推演提案】

**方案标题**: "适应辐射"战略转型

**来源定理**: 进化生物学 - 关键创新与适应辐射

**映射逻辑**: AI 冲击不是威胁，而是环境剧变带来的新生态位机会。SaaS 企业需要识别自己的"关键创新"（不可替代资产），通过适应辐射占领新的生态位。

**可执行方案**:

1. **识别关键创新资产**
   - 数据资产：客户历史数据是 AI 无法复制的
   - 行业 know-how：垂直领域深度知识
   - 工作流集成：与企业现有系统的深度绑定

2. **生态位跳跃策略**
   - 从"工具提供商" → "AI 增强工作流 orchestrator"
   - 不是与 AI 竞争，而是用 AI 增强自身
   - 新定位："带着你的数据，我们用 AI 帮你更高效"

3. **适应辐射产品线**
   - 核心产品：AI 增强版（原有功能 + AI 辅助）
   - 新产品 1：数据资产变现（分析、洞察、预测）
   - 新产品 2：AI 工作流编排（打通多个 AI 工具）
   - 新产品 3：垂直领域 AI 训练（用客户数据 fine-tune）

**预期效果**:
- 从"被替代者"转变为"AI 时代的底座"
- 收入结构多元化，降低单一产品依赖
- 建立 AI 时代的护城河（数据 + know-how）

### 【伴随解】- Adjoint Balancer

**成本-结构分析**:

| 方案 | 时间成本 | 资金成本 | 机会成本 | 可行性评分 |
|------|----------|----------|----------|------------|
| AI 增强版 | 3个月 | 中 | 低 | ⭐⭐⭐⭐⭐ |
| 数据变现 | 6个月 | 高 | 中 | ⭐⭐⭐⭐ |
| AI 工作流编排 | 6个月 | 高 | 高 | ⭐⭐⭐ |

**最优落地路径**:
1. **立即执行**: AI 增强版 - 快速响应，止血流失
2. **3个月后**: 启动数据变现项目
3. **持续评估**: AI 工作流编排的 ROI

## 示例 4：商业模式看不穿 (新增领域：人类学)

**用户输入**: "我创业 3 年了，产品还行，但总感觉商业模式不对，看不穿本质。"

### 【范畴骨架】- Domain A

| 类型 | 元素 | 说明 |
|------|------|------|
| Object | 产品 | 具体的服务/商品 |
| Object | 用户 | 购买者/使用者 |
| Object | 收入 | 盈利来源 |
| Object | 创始人 | 决策者，有认知盲区 |
| Morphism | 交易 | 产品换取收入 |
| Morphism | 价值创造 | 产品解决用户问题 |
| Morphism | 价值捕获 | 将价值转化为收入 |
| Constraint | 认知盲区 | 创始人身在其中，难以看清 |
| Constraint | 路径依赖 | 3年积累形成的惯性 |

### 【异构域】- Domain B: anthropology (人类学) ⭐ v2.5 新增

**选择理由**: 商业模式的本质是"人类行为模式"，人类学的"参与观察"和"田野调查"方法可以提供外部视角。

**Domain B 核心结构** (来自 anthropology_v2.md):

**Core Objects**:
- 文化 (Culture): 共享的意义系统
- 实践 (Practice): 日常行为模式
- 符号 (Symbol): 承载意义的载体

**Core Morphisms**:
- 田野调查 (Fieldwork): 深度浸入式研究
- 参与观察 (Participant Observation): 既是参与者又是观察者
- 他者化 (Othering): 将熟悉变为陌生，获得新视角

**关键定理**:
- **定理 1: 陌生化原则** - 将熟悉的事物视为陌生，才能看清其结构
  - Applicable_Structure: 创始人陷入日常运营，难以看清商业模式本质
  - Mapping_Hint: 当 Domain A 面临"看不穿"问题时，通过他者化视角，将自己从参与者转化为观察者，剥离情感与惯性，看清真实结构
  - Case_Study: 人类学家进入异文化，先悬置自己的文化预设

### 【映射矩阵】

| Domain A | 映射关系 | Domain B | 同构性验证 |
|----------|----------|----------|------------|
