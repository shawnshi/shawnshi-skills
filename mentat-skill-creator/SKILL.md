# Codex 技能创建与修订

## 核心规则

- 将技能限定为一个清晰、可复用的工作。
- 把 Codex 已知的通用知识留在模型中，只写领域规则、脆弱步骤、工具契约和验证方法。
- 使用命令式步骤，明确输入、输出、权限边界和失败处理。
- `SKILL.md` frontmatter 只包含 `name` 和 `description`。
- `name` 使用小写字母、数字和连字符，不超过 64 个字符，并与目录名一致。
- `description` 同时说明能力和触发场景；不要把触发条件只写在正文中。
- 不创建、读取或维护 `skill.json` 作为技能来源。
- 不要求输出隐藏推理，也不假设不存在的工具名或运行时协议。
- 不硬编码用户目录、其他运行时目录或机器专属绝对路径。

## 执行流程

1. 理解用途：从用户请求和现有技能中提取具体调用示例、非目标、输入、输出与风险。
2. 检查现状：完整阅读目标 `SKILL.md`，列出相关的 `scripts/`、`references/`、`assets/` 和 `agents/openai.yaml`。保留有效资源，避免重复内容。
3. 规划边界：决定哪些指令保留在正文，哪些细节按需放入资源文件。正文保持精简，优先少于 500 行。
4. 创建或修改：
   - 新技能先确认目标目录；
   - 现有技能只修改用户授权的文件；
   - 使用补丁式编辑，保留无关用户改动；
   - 资源引用使用相对技能根目录的路径，并尽量保持一层引用。
5. 设计权限：只读任务默认不产生外部写入；发布、发送、持久化、删除、安装依赖和配置修改需要明确授权。
6. 设计工具：使用当前环境真实提供的能力。只有当任务能拆成独立工作面且并行有明显收益时，才使用子代理。
7. 验证：使用系统技能校验脚本时，先确认脚本可用；否则至少检查 frontmatter、目录名、描述长度、相对链接、行数和禁用模式。
8. 对复杂技能做前向测试时，给测试代理最少任务上下文和原始制品，不泄露预期答案或修改意图。测试可能产生外部副作用时先取得授权。
9. 汇报变更文件、验证结果和残余风险。不自动把技能登记到知识库或其他系统。

## 标准结构

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml          # 可选，界面元数据
├── scripts/                 # 可选，确定性或重复执行逻辑
├── references/              # 可选，按需读取的领域资料
└── assets/                  # 可选，输出使用的模板或素材
```

```markdown
---
name: skill-name
description: 说明技能做什么，以及用户在什么情况下应使用它。
---

# 技能标题

## 执行流程

1. [命令式步骤]

## 完成检查

- [可验证条件]
```

## 静态检查

- frontmatter 只有两个必需字段。
- 名称与目录匹配，描述包含明确触发语境。
- 正文没有机器专属绝对路径、过时运行时名或虚构工具。
- 子代理为条件化能力，不是每次执行的前置条件。
- 删除、发布、持久化、依赖安装和配置修改均有授权边界。
- 所有相对资源链接存在，脚本参数来自实际帮助或源码。
