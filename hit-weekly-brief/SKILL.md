# 数字健康周报

## 执行流程

1. 确定出刊日期、报告周期、截止时点、地区、受众和决策重点。用户指定的日期优先；用户未指定时采用当前自然周的周一至周日，出刊日期取该周周日。若生成时本周尚未结束，只检索到当前时点，并在标题下明确截止时间和“本周尚未结束”。仅当用户明确要求“上一完整周”或“最近一个完整自然周”时，才采用已经结束的上一自然周。
2. 若任务延续既有周报归档，先只读检查最近三期文件名和标题。既有、稳定且与用户要求不冲突的出刊惯例优先于通用默认值；无法确认时明确说明采用的日期口径。
3. 选择与任务有关的检索面：战略研究、监管政策、技术落地、投融资与市场、跨行业参考。
4. 检索当前资料。优先监管机构、政府部门、论文和会议原文、公司公告、研究机构原文及可验证数据集。
5. 同时记录发布时间与事件日期，避免把旧报告的二次传播列为本周新信号。事件日期必须位于报告起始日与截止时点之间；仅作背景引用的旧事件需明确标为背景，不得列作本周事件。
6. 合并转载、摘要和同一报告的不同入口。对高影响结论尽量用独立来源复核。
7. 分开书写：
   - 已核实事实；
   - 来源机构的主张；
   - 基于证据的分析判断；
   - 尚待验证的假设。
8. 不强求每周都出现反共识信号。只有证据与主流判断形成真实冲突时，才列出分歧。
9. 当存在三个以上独立检索面且环境支持并行工作时，可并行检索；否则直接完成。
10. 生成报告并附来源。默认不保存、不发布、不写入知识库；这些动作需用户明确授权。

## 保存与命名

- 用户明确要求保存时，优先使用用户指定目录；未指定目录且存在稳定归档位置时沿用该位置，不另建平行目录。
- 默认文件名为 `DHWB-YYYYMMDD.md`，其中日期是当期周日的出刊日期，也是报告周期结束日，不是任意生成日或上一周结束日。
- 保存前检查目标是否已存在。没有覆盖授权时保留原文件，并使用用户认可的区分方式；不得静默覆盖。
- 文件名、标题周期、出刊日期和截止说明属于同一语义单元。用户修正其中任一项时，重新检查其余字段，禁止只做局部改名。

## 输出要求

完整周报按需读取 [references/template.md](references/template.md)。简短请求不必机械填满模板，可使用以下结构：

1. 本周结论
2. 关键事件与来源
3. 政策和市场影响
4. 技术落地信号与证据限制
5. 对医疗信息化业务的传导影响
6. 建议动作与验证节点
7. 信息缺口

每条关键事件至少包含事件日期、主体、动作、影响判断和直接来源。对论文、政策和研究报告优先链接原文，不链接搜索结果页。

`assets/` 下的任务文件只能用于扩展检索面；[examples/DHWB-Reference.md](examples/DHWB-Reference.md) 仅作格式示例，不能作为事实来源。

## 验证门禁

保存前必须满足以下断言：

- 报告起始日为周一，结束日为同一自然周的周日，出刊日期等于结束日。
- 文件名中的日期等于出刊日期；标题同时包含正确的起止日期。
- 生成日期位于报告周期内；生成时周日尚未结束或结束日仍在未来时，正文显式标注截止时点和周期未完结状态。
- 本周事件日期不早于起始日、不晚于截止时点；旧事件只能作为背景。
- 改名、改周期或刷新内容后，重新执行全部断言。

环境提供 Python 3 且报告已保存时，运行确定性校验器：

```powershell
python scripts/validate_weekly_brief.py --file <report.md> --period-start YYYY-MM-DD --period-end YYYY-MM-DD --issue-date YYYY-MM-DD --cutoff <ISO-8601时间>
```

如用户明确指定非标准文件名，附加 `--allow-custom-filename`。校验器不可用时按同一断言人工检查，并如实说明未运行原因。

## 完成检查

- 报告周期与事件日期一致。
- 文件名、出刊日期、标题周期和截止时点一致。
- 关键结论均有可访问来源。
- 事实、来源观点和分析推断已分层。
- 没有把数据缺口隐藏成确定性断言。
- 建议动作包含适用条件或验证节点。
